========================================================
  Vision_Analysis_agent 项目优化分析报告
========================================================

一、高危问题（需尽快修复）
==========================

【1】thread_id 硬编码 — api/main.py:37
--------------------------------------
所有用户共享 "user_session_001"，会导致 LangGraph 检查点状态混乱，
不同用户互相覆盖。

修复方案：
    # 改用用户标识生成唯一 thread_id
    thread_id = f"{username}_{uuid.uuid4().hex[:8]}"


【2】上传文件从不删除 — api/main.py:30
--------------------------------------
每次请求的临时图片永久保留在 uploads/ 目录，磁盘会持续增长。
需在 finally 块中确保文件清理。

修复方案：
    try:
        full_state = agent.run(...)
        ...
    finally:
        if os.path.exists(file_path):
            os.remove(file_path)


【3】AnalysisAgent.self.db 未初始化 — analysis_agent.py
---------------------------------------------------------
_save_node 方法用到了 self.db，但 __init__ 从未初始化它，
调用时直接抛出 AttributeError。该节点也没有加入 workflow，
是一段死代码，建议直接删除。


【4】CORS 配置危险组合 — api/main.py:16
----------------------------------------
allow_origins=["*"] 与 allow_credentials=True 同时使用是不安全组合
（浏览器会拒绝，且有 CSRF 风险）。

修复方案：
    # 生产环境将 allow_origins 限制为具体域名
    allow_origins=["https://your-domain.com"]
    # 或关闭 credentials
    allow_credentials=False


【5】文件上传无安全校验 — api/main.py:27
-----------------------------------------
问题清单：
- 无 MIME 类型验证（仅凭扩展名判断，可被伪造）
- 无文件大小限制（可导致磁盘溢出）
- filename 未过滤 ../ 路径遍历字符

修复方案：
    ALLOWED_MIME = {"image/jpeg", "image/png", "image/webp"}
    MAX_SIZE = 10 * 1024 * 1024  # 10MB

    content = await image.read()
    if len(content) > MAX_SIZE:
        return {"status": "error", "message": "文件过大"}
    if image.content_type not in ALLOWED_MIME:
        return {"status": "error", "message": "不支持的文件类型"}


二、中危问题（影响稳定性/性能）
================================

【6】LLM 每次都重新实例化 — get_llm.py
----------------------------------------
get_llm() 每次调用创建新对象，AnalysisAgent.__init__ 调用两次 get_llm()，
应做成单例或缓存。

修复方案：
    _llm_instance = None

    def get_llm():
        global _llm_instance
        if _llm_instance is None:
            _llm_instance = ChatTongyi(model="qwen-plus", temperature=0)
        return _llm_instance


【7】analyze_image 返回类型标注错误 — vision_agent.py:15
----------------------------------------------------------
方法标注为 -> str，实际返回 VisionResponse 对象，会误导类型检查工具。

修复方案：
    def analyze_image(self, image_path: str) -> VisionResponse:


【8】异步接口里使用同步 IO — api/main.py:30
--------------------------------------------
FastAPI 异步路由中用 shutil.copyfileobj 同步写文件，会阻塞事件循环。

修复方案：
    import aiofiles

    async with aiofiles.open(file_path, "wb") as f:
        content = await image.read()
        await f.write(content)


【9】sys.path.append 反模式 — user_db.py:9、canteen_db.py:8
-------------------------------------------------------------
用 sys.path 黑魔法导入同级模块，脆弱且不可维护。

修复方案：
    # 改为相对导入
    from ..get_embed import DashScopeEmbeddingAdapter


【10】两套数据库实现并存 — database_utils/
-------------------------------------------
Chroma 版（user_db.py / canteen_db.py）和 SQLite 版
（user_db_sql.py / canteen_db_sql.py）同时存在，
不清楚哪个在实际使用，容易导致维护混乱和行为不一致。

建议：明确选用一套方案，删除另一套，或用适配器模式统一接口。


【11】数据库连接未关闭 — user_db_sql.py、canteen_db_sql.py
-----------------------------------------------------------
SQLite 连接在 __init__ 中打开，没有 close() 或 __del__ 方法，
属于连接泄漏。

修复方案：
    def close(self):
        if self.conn:
            self.conn.close()

    def __del__(self):
        self.close()

    # 或实现上下文管理器
    def __enter__(self): return self
    def __exit__(self, *args): self.close()


【12】环境变量未校验 — get_llm.py:4
--------------------------------------
load_dotenv 后没有检查 DASHSCOPE_API_KEY 是否存在，
API Key 缺失时报错信息不直观。

修复方案：
    import os
    load_dotenv(find_dotenv())
    if not os.getenv("DASHSCOPE_API_KEY"):
        raise ValueError("环境变量 DASHSCOPE_API_KEY 未设置，请检查 .env 文件")


【13】API 响应格式不统一 — api/main.py
----------------------------------------
当前三种响应格式不一致，前端处理麻烦：
- 成功：      {"status": "success", "data": ...}
- 无效图片：  {"status": "invalid_image", "message": ...}
- 异常：      {"status": "error", "message": ...}

修复方案（在 schemas.py 中统一定义）：
    class ApiResponse(BaseModel):
        status: str
        message: str = ""
        data: Optional[dict] = None
